# Documentation Principles

Documentation in this repository is the **canonical** source for Axionax Protocol Core — the primary reference for developers, operators, and audit.

---

## 1. Language

- **Use English for all documentation.** All docs in `docs/`, `core/docs/`, root `*.md`, and linked references should be written in English. Technical terms (e.g. chain_id, RPC, staking) remain as-is.

---

## 2. Role of docs in this repo

- **Canonical:** Documents in `docs/`, `core/docs/`, `README.md`, and related `.md` files are the source of truth for the protocol (core), not marketing or promotional material.
- **Goal:** Provide accurate, complete, and citable information for development, deployment, and verification.

---

## 3. Tone and style

| Principle | Practice |
|-----------|----------|
| **Formal** | Use formal, clear, and direct language. Avoid colloquialisms or unnecessary slang. |
| **Somewhat academic** | Use correct technical terminology (blockchain, consensus, RPC, staking). Define or reference terms when needed. No need for full research-paper style. |
| **Substance over aesthetics** | Prioritise correctness and completeness. Do not prioritise visual or narrative flair. |

---

## 4. What to do

- Use consistent headings and numbering (e.g. 1. 2. 3. or ## ###).
- Use tables for comparisons or summaries (config, spec, checklist).
- Reference file paths in the repo when referring to config, scripts, or other docs.
- Specify versions or conditions when relevant (e.g. chain_id, Rust/Python version).

---

## 5. What to avoid

- Relying on emoji or graphics for effect (minimal use for emphasis is acceptable).
- Long promotional or marketing text in technical docs.
- Predominantly casual or colloquial tone.

---

*This document defines the documentation principles for axionax-core-universe: canonical, formal, and citable. All documentation is in English.*
